## 1. 비즈니스 프로세스 상세

### 1.1 전체 서비스 플로우

```
┌───────────────────────────────────────────────────────────────────┐
│                        CasasTrade 서비스                          │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  [크롤링] → [상품 노출] → [입찰] → [낙찰] → [정산] → [입고]     │
│     │                                                │            │
│     │                                         [감정] ← [WMS]     │
│     │                                                │            │
│     │                                         [수선] ← [업체연동]│
│     │                                                │            │
│     │                                         [출고] → [배송]     │
│     │                                                             │
│  [시세 크롤링] → [시세 조회]                                      │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

### 1.2 입찰-정산 상세 프로세스

#### Step 1: 크롤링 수집

```
1. 크롤러가 일본 경매 사이트에서 상품 데이터 수집
2. crawled_items 테이블에 저장 (UPSERT by item_id + auc_num)
3. Elasticsearch 인덱스 동기화
4. 이미지 비동기 다운로드 및 처리 (800x800, JPEG 80%)
5. 필터 설정에 따라 is_enabled 상태 결정
```

#### Step 2: 사용자 입찰

```
1. 사용자가 상품 조회 (필터, 검색, 정렬)
2. 입찰가 입력 (경매장별 단위 검증)
3. 예치금 잔액 확인
   - 개인: deposit_balance ≥ 차감 예정액
   - 법인: daily_limit - daily_used ≥ 차감 예정액
4. 예치금 차감 (입찰 시점)
5. Google Sheets 동기화
6. KakaoTalk 알림 발송 (해당 상품 기존 입찰자에게)
```

#### Step 3: 낙찰 확인 및 정산

```
1. 관리자가 낙찰 여부 확인 (수동 또는 크롤링 업데이트)
2. 입찰 상태 → 'completed' 변경
3. 낙찰가(winning_price) 기록
4. 수수료 계산:
   a. 현지 수수료 (경매장별)
   b. 관세 (8%, 20만원 이하 면제)
   c. 부가세 (10%)
   d. 사용자 커미션
5. daily_settlements 레코드 생성/업데이트
6. 미낙찰 시: 예치금 환불
```

#### Step 4: 정산 결제

```
1. 사용자에게 정산 금액 안내
2. 입금 방법:
   a. 예치금 자동 차감 (잔액 충분 시)
   b. 은행 송금 → Popbill 자동 매칭
   c. 수동 확인 (관리자)
3. 정산 상태: unpaid → pending → paid
```

#### Step 5: WMS 입고 및 처리

```
1. 낙찰 확인 시 WMS 자동 등록
2. 국내 도착 → DOMESTIC_ARRIVAL_ZONE 스캔
3. 감정 요청 시 → 감정서 발행
4. 수선 필요 시:
   a. 내부 수선 → INTERNAL_REPAIR_ZONE
   b. 외부 수선 → EXTERNAL_REPAIR_ZONE → 업체 시트 동기화
5. 수선 완료 → REPAIR_DONE_ZONE
6. 출고 준비 → OUTBOUND_ZONE
```

#### Step 6: 배송

```
1. 배송 정보 입력 (수취인, 주소)
2. Logen 송장번호 발급
3. Logen 주문 등록
4. 배송 라벨 PDF 생성 (100mm×150mm)
5. 배송 추적 (API 자동 조회)
6. 배송 완료 확인
```

### 1.3 예치금 시스템

#### 계정 유형

| 항목      | 개인 (individual)          | 법인 (corporate)           |
| --------- | -------------------------- | -------------------------- |
| 잔액 관리 | `deposit_balance` (충전식) | `daily_limit` (한도 방식)  |
| 충전      | 은행 송금 → 관리자 승인    | 한도 설정 (관리자)         |
| 차감      | 입찰 시 잔액에서 차감      | 당일 사용액에 추가         |
| 환불      | 미낙찰/취소 시 잔액 복원   | 당일 사용액에서 차감       |
| 초기화    | -                          | 매일 `daily_used = 0` 리셋 |

#### 자동 입금 확인 (Popbill)

```
1. 사용자가 충전 요청 (금액 + 입금자명 입력)
2. deposit_transactions 레코드 생성 (status: pending)
3. cron이 주기적으로 Popbill EasyFinBank API 호출
4. 은행 거래 내역에서 매칭:
   - 입금자명 일치
   - 금액 일치
   - 아직 사용되지 않은 거래
5. 매칭 성공 → 자동 승인 → 예치금 반영
6. used_bank_transactions에 사용 기록
7. 최대 재시도: 12회
```

### 1.4 수수료 계산 상세

#### 계산 공식

```
[1] 현지수수료 = f(낙찰가, 경매장)
    - EcoAuc: 티어별 차등 (50만엔 이하 10%, 100만엔 이하 9%, ...)
    - BrandAuc: 고정 ¥2,200
    - 기타: 경매장별 규칙

[2] 과세가격 = (낙찰가 + 현지수수료) × 환율 × 0.6

[3] 관세 = 과세가격 × 8%
    - 20만원 이하: 면세

[4] 부가세 = (과세가격 + 관세) × 10%
    - 관세 면세 시: 과세가격 × 10%

[5] 커미션 = (낙찰가 × 환율) × 커미션비율
    - 사용자별 개별 설정 가능

[6] 총 비용 = (낙찰가 + 현지수수료) × 환율 + 관세 + 부가세 + 커미션
```

## 2. 감정 시스템 (CAS System)

### 2.1 감정 유형

| 유형             | 설명                | 필수 정보        | 비용             |
| ---------------- | ------------------- | ---------------- | ---------------- |
| **quicklink**    | URL 기반 원격 감정  | 상품 URL, 플랫폼 | 크레딧 차감      |
| **offline**      | 실물 방문/택배 감정 | 배송 정보        | 유료 (38,000원~) |
| **from_auction** | 옥션 낙찰 상품 감정 | 입찰 ID          | 정산에 포함      |

### 2.2 감정 결과

| 결과        | 설명         |
| ----------- | ------------ |
| `authentic` | 정품 확인    |
| `fake`      | 가품 판별    |
| `uncertain` | 판별 보류    |
| `pending`   | 감정 대기 중 |

### 2.3 감정서 발급

```
1. 감정 완료 → 인증번호 생성
2. QR 코드 생성 (인증번호 + 접근키)
3. PDF 감정서 생성:
   - pdf-lib + NotoSansKR 한국어 폰트
   - 브랜드 정보, 모델명, 시리얼
   - 감정 결과 + 감정일
   - QR 코드 임베디드
   - 감정 이미지
4. S3 업로드
5. 공개 조회 URL: /appr/result/{certificateNumber}
```

| 티어                | 설명        | Quick Link    |

## 3. WMS (창고 관리 시스템)

### 3.1 창고 구역 및 흐름

```
[국내 도착] ──스캔──> DOMESTIC_ARRIVAL_ZONE
    │
    ├── 감정 필요 → (감정 처리)
    │
    ├── 내부 수선 → INTERNAL_REPAIR_ZONE
    │                    │
    │                    └── 완료 → REPAIR_DONE_ZONE
    │
    ├── 외부 수선 → EXTERNAL_REPAIR_ZONE
    │                    │
    │                    └── 완료 → REPAIR_DONE_ZONE
    │
    ├── 보류 사유 → HOLD_ZONE
    │
    └── 출고 준비 → OUTBOUND_ZONE
                        │
                        └── 배송 → (배송 관리)
```

### 3.2 바코드 체계

| 바코드 유형        | 형식                        | 예시            |
| ------------------ | --------------------------- | --------------- |
| `item_uid`         | `{소스}-{item_id}-{bid_id}` | `ECO-12345-678` |
| `external_barcode` | 경매장 원본                 | -               |
| `internal_barcode` | 내부 관리용                 | -               |

### 3.3 스캔 이벤트

모든 물품 이동은 `wms_scan_events` 테이블에 기록됩니다:

- 스캐너 대상: 바코드 (external 또는 internal)
- from_location → to_location
- 이전 상태 → 다음 상태
- 담당자, 메모

## 4. 수선 관리 시스템

### 4.1 수선 케이스 상태

```
ARRIVED → DRAFT → PROPOSED → ACCEPTED → DONE
                       ↘ REJECTED
```

| 상태       | 설명                    |
| ---------- | ----------------------- |
| `ARRIVED`  | 물품 도착 (수선 대기)   |
| `DRAFT`    | 수선 견적 작성 중       |
| `PROPOSED` | 고객에게 수선 제안 발송 |
| `ACCEPTED` | 고객 수락 → 수선 진행   |
| `REJECTED` | 고객 거절               |
| `DONE`     | 수선 완료               |

### 4.2 수선 업체 연동

Google Sheets를 통해 수선 업체와 데이터를 공유합니다:

| 업체         | 시트 연동 | 설명           |
| ------------ | --------- | -------------- |
| 리리         | 개별 시트 | 외부 수선 업체 |
| 크리뉴       | 개별 시트 | 외부 수선 업체 |
| 성신사(종로) | 개별 시트 | 외부 수선 업체 |
| 연희         | 개별 시트 | 외부 수선 업체 |
| 까사(내부)   | 내부 시트 | 내부 수선팀    |

### 4.3 수선 비용 처리

- 수선 비용은 `repair_fee` 필드에 원화(VAT 포함)로 기록
- 정산 시 `repair_fee`, `repair_vat` 별도 집계
- `daily_settlements.repair_count`에 수선 건수 포함

## 5. 배송 시스템

### 5.1 로젠 택배 연동

| 단계 | API                        | 설명                  |
| ---- | -------------------------- | --------------------- |
| 1    | `getContractInfo()`        | 계약 정보 확인        |
| 2    | `getDeliveryInfo(zipcode)` | 우편번호→배송점 매핑  |
| 3    | `issueSlipNumber()`        | 송장번호 발급         |
| 4    | `registerOrder(params)`    | 주문 등록 (배송 시작) |
| 5    | `trackShipment(slipNo)`    | 배송 추적 조회        |

### 5.2 배송 상태

```
ready → slip_issued → order_registered → picked_up
→ in_transit → out_for_delivery → delivered
                               ↘ failed / returned
```

### 5.3 배송 라벨

| 형식      | 크기           | 용도      |
| --------- | -------------- | --------- |
| 표준 라벨 | 100mm × 150mm  | 개별 발송 |
| A4 시트   | A4 (복수 라벨) | 일괄 인쇄 |

## 8. 보안 사항

| 영역          | 구현                                         |
| ------------- | -------------------------------------------- |
| 비밀번호      | SHA-256 해시                                 |
| 세션          | HttpOnly, SameSite=lax, Secure(prod)         |
| CORS          | `ALLOWED_ORIGINS` 화이트리스트               |
| 관리자 접근   | 세션 기반 + 메뉴별 권한 + 활동 로깅          |
| API 인증      | 세션 미들웨어 (`isAuthenticated`, `isAdmin`) |
| SQL Injection | `mysql2` 파라미터 바인딩                     |
| 파일 업로드   | multer (크기 제한, MIME 타입 검사)           |
| 환경 변수     | `.env` 파일 (서비스 키, 비밀번호)            |
| QR 접근키     | 감정서별 고유 `qr_access_key` (32자)         |
